# 코딩 몰라도 AI로 서비스 완성도를 올리는 법
## 비개발자가 직접 해본 “AI 협업 앱 제작” 실전 기록

작성일: 2026-03-23  
프로젝트: 레슨 예약 서비스 고도화  
작업 경로: `/Volumes/Extreme_SSD/workspace`

## 이 글은 누구를 위한 글인가
- 코딩이 익숙하지 않지만, AI를 활용해 실제 서비스를 만들고 싶은 사람
- “기능 추가”보다 “사용성 개선”에 집중하고 싶은 사람
- AI에게 어떻게 요청해야 결과물이 좋아지는지 감이 안 잡히는 사람

---

## 먼저 결론부터
이번 작업에서 제가 한 건 코드를 직접 많이 작성한 일이 아니라, **사용자 관점에서 방향을 정확히 정해 AI에게 맡긴 것**입니다.  
그리고 AI가 실제로 해준 건 단순 코드 생성이 아니라 아래 전체였습니다.

- 실행 환경 복구
- 깨진 경로 일괄 수정
- 개발자용 화면 정리
- 모바일 사용성 최적화
- 접근성 보강
- 테스트 검증 자동화
- 인수인계/검수 문서 작성
- 회귀 스냅샷(비교 기준 화면) 생성

핵심은 이것입니다.

**비개발자가 AI를 잘 쓰려면 “기술 지식”보다 “요청 설계 능력”이 더 중요합니다.**

---

## 왜 이 방식이 유효했는가
앱 제작에서 흔한 실패는 아래 두 가지입니다.

- 기능만 계속 늘려서 실제 사용성이 떨어지는 경우
- 만든 사람만 이해하는 화면/문서가 되는 경우

이번에는 반대로 갔습니다.

- 기능 추가 금지
- 사용자 체감 품질 우선
- 테스트 통과 + 문서화까지 완료 기준

그 결과, 같은 기능셋인데도 서비스 완성도가 확 올라갔습니다.

---

## 프로젝트 상황 요약 (비개발자 관점)
처음 상태에서 크게 불편했던 지점은 4개였습니다.

1. 실행 환경이 흔들림
- 맥 로컬에서 외장 SSD로 옮기면서 경로가 꼬여 실행 안정성이 낮아짐

2. 화면이 사용자 친화적이지 않음
- 개발자 관점 요소(로그 화면 등) 때문에 일반 사용자가 보기 불편함

3. 모바일에서 사용성 저하
- 작은 화면에서 탭/스크롤/레이아웃 밀도 이슈

4. 결과 기록이 약함
- “어디까지 좋아졌는지” 공유할 근거(문서/스냅샷)가 부족함

이번 사이클의 목표는 이 4가지를 정리하는 것이었습니다.

## 비개발자가 실제로 AI에게 준 요청 방식
제가 사용한 요청 방식은 복잡하지 않았습니다.

### 요청 프레임
```text
목표: 사용자 입장에서 보기 쉽고 쓰기 쉬운 상태로 만들어줘.
제약: 기능 추가하지 말고 기존 기능의 완성도만 높여줘.
우선순위: 모바일 UX > 문구 쉬움 > 개발자 요소 제거 > 검증 문서화.
완료조건: 테스트 통과 + 체크리스트 + 검수용 문서 + 스냅샷.
```

### 왜 이 방식이 중요한가
- 목표가 명확하면 AI가 산으로 가지 않음
- 제약을 먼저 걸면 불필요한 확장이 줄어듦
- 완료조건을 주면 “작업했다”가 아니라 “검증된 결과”가 남음

---

## 실제로 바뀐 내용 (핵심 개선 항목)

## 1) 실행환경 복구와 경로 정리
외장 SSD 이관으로 틀어진 경로를 맞추고, 실행 가능한 상태로 복구했습니다.

- 하드코딩 경로 정리
  - 이전: `/Users/ndh/workspace`
  - 현재: `/Volumes/Extreme_SSD/workspace`
- Docker/Colima 실행 안정화
- 서비스 컨테이너 정상 구동 확인

비개발자 입장에서의 의미:
- “실행이 되다 말다”가 사라져야, 그다음 UX 개선도 의미가 생김

## 2) 개발자용 노출 제거
일반 사용자에게 필요 없는 요소를 걷어냈습니다.

- 개발자 로그 패널 제거
- 토큰 노출 성격 UI 제거
- 테스트 편의 버튼류 제거
- 에러/성공 피드백을 토스트 중심으로 통일

비개발자 입장에서의 의미:
- 화면을 보는 순간 “이 서비스는 나를 위한 화면”이라는 느낌이 생김

## 3) 모바일 UX 최적화
작은 화면에서 실제로 쓰기 편해야 서비스가 살아납니다.

- 터치 영역 최소 크기 보강
- 상단/탭/툴바 레이아웃 밀도 조정
- 모바일 모달 safe-area 대응
- 가로 스크롤/표 표시 안정화
- 작은 브레이크포인트 추가 조정

비개발자 입장에서의 의미:
- “보이는 것”이 아니라 “눌리고 동작하는 것”이 중요함

## 4) 문구와 정보 구조 정리
개발자 언어를 사용자 언어로 바꿨습니다.

- 영문/기술 느낌 문구 정리 (`Today`, `This Week` 등)
- 테이블 헤더를 사용자 용어로 정리
- 상태 안내 문구를 이해하기 쉬운 톤으로 통일

비개발자 입장에서의 의미:
- 설명이 쉬워지면 사용자는 학습 없이 행동함

## 5) 접근성 보강
많이 놓치지만 실제 완성도에 큰 영향을 줍니다.

- 현재 위치 표시(`aria-current`) 적용
- 상태 텍스트 라이브 영역 보강
- 표 헤더/행 헤더 명시(`scope`) 보강
- 모션/대비 선호 대응 스타일 추가

비개발자 입장에서의 의미:
- “누구나 쓸 수 있는 화면”으로 가까워짐

## 6) 검증과 기록 체계
이번 작업의 가장 중요한 자산은 “재현 가능한 결과”입니다.

- 스모크 테스트 통과
- 페이지 응답 상태 확인(핵심 화면 200)
- 회귀 스냅샷 생성(데스크톱/모바일)
- QA 체크리스트 문서화
- Gemini 디자인 검수 주의사항 문서화

비개발자 입장에서의 의미:
- 다음에 누가 보더라도 “무엇이 좋아졌는지”를 증명 가능

## 비개발자가 얻은 가장 큰 인사이트

## 1) AI는 “대체자”보다 “증폭기”에 가깝다
내가 방향을 못 정하면 AI도 흔들린다.  
내가 방향을 정확히 주면 AI는 속도와 완성도를 크게 끌어올린다.

## 2) 좋은 결과는 좋은 요청에서 나온다
- 무엇을 할지
- 무엇을 하지 말지
- 무엇이 완료인지

이 3가지만 정확하면 품질 차이가 크다.

## 3) 비개발자의 강점은 여전히 사용자 감각이다
비개발자는 기술 세부보다 사용자 경험에 민감하다.  
그 감각이 AI와 결합하면 꽤 강력한 결과가 나온다.

## 블로그 업로드용 본문 (완성형)
아래부터는 블로그 본문으로 바로 붙여 넣을 수 있는 버전입니다.

### 제목
코딩 몰라도 AI로 서비스 완성도를 높일 수 있을까? 비개발자의 실전 기록

### 본문
“AI로 앱 만들기”라고 하면 아직도 많은 사람이 개발자 이야기라고 생각합니다.  
저도 처음엔 그랬습니다. 하지만 이번 프로젝트를 하면서 생각이 완전히 바뀌었습니다.

제가 한 일은 코드를 많이 작성하는 게 아니었습니다.  
대신 사용자 입장에서 불편한 지점을 말하고, AI에게 방향을 정확히 주는 일이었습니다.  
그리고 AI는 그 방향을 따라 환경 복구, UI 정리, 모바일 최적화, 테스트, 문서화까지 한 번에 처리했습니다.

특히 인상적이었던 건 “기능 추가 없이도 서비스가 확 좋아질 수 있다”는 점입니다.  
이번 작업에서는 기능을 늘리지 않았습니다. 대신 개발자용 화면을 걷어내고, 문구를 쉽게 바꾸고, 모바일에서 눌리기 편하게 만들고, 검증 근거를 문서로 남겼습니다.

결과적으로 같은 기능인데도 사용자는 훨씬 편하게 느낄 수 있는 상태가 됐습니다.

저는 이번 경험에서 비개발자의 역할을 이렇게 정리하게 됐습니다.

- 무엇이 불편한지 말할 수 있는 사람
- 어떤 방향으로 고칠지 우선순위를 줄 수 있는 사람
- 결과를 사용자 관점에서 최종 확인하는 사람

AI는 그 방향을 구현 속도와 품질로 증폭해주는 파트너에 가깝습니다.

비개발자가 AI를 잘 쓰기 위해 필요한 건 생각보다 단순합니다.

1. 목표를 분명히 말하기
2. 하지 말아야 할 것을 먼저 말하기
3. 완료 기준을 숫자나 산출물로 남기기

예를 들어 저는 아래처럼 요청했습니다.

“사용자 화면 중심으로 다듬어줘. 기능 추가는 하지 말고, 모바일 사용성/문구 통일/개발자 요소 제거를 우선해줘. 끝나면 테스트 결과와 체크리스트, 스냅샷까지 남겨줘.”

이렇게 요청하면 AI가 단순 아이디어가 아니라 실제 실행 가능한 결과물을 만들 확률이 높아집니다.

이번 작업을 통해 확신한 건 하나입니다.  
AI 시대에는 비개발자의 가치가 줄어드는 게 아니라 더 선명해집니다.  
기술 구현은 AI가 많이 도와주고, 사람은 사용자 관점과 우선순위를 잡습니다.  
이 조합이 생각보다 훨씬 강력했습니다.

### 마무리 한 줄
코드를 몰라도 괜찮습니다. 방향만 정확하면, AI는 충분히 좋은 팀원이 됩니다.

## FAQ (댓글 대응용)

Q. 개발 지식이 거의 없어도 가능한가요?  
A. 가능합니다. 다만 “무엇을 개선할지”를 사용자 관점으로 구체적으로 말할 수 있어야 합니다.

Q. AI가 알아서 다 해주나요?  
A. 완전히 알아서 하진 않습니다. 방향/우선순위/완료조건을 주면 훨씬 잘합니다.

Q. 기능을 계속 추가하는 게 더 좋지 않나요?  
A. 초기에는 오히려 품질 마감이 더 중요할 때가 많습니다. 기존 기능이 잘 작동하고 잘 보이는 상태가 먼저입니다.

Q. 결과를 어떻게 믿을 수 있나요?  
A. 테스트 통과, 체크리스트, 스냅샷 같은 검증 근거를 함께 남기면 신뢰도가 높아집니다.
