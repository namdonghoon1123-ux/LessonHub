# AI 협업 운영안 (2026-04-02)

이 문서는 LessonHub 레포에서 Codex와 Claude Code가 같이 작업할 때 충돌을 줄이고 속도를 높이기 위한 운영 아이디어를 정리한 문서입니다.

## 1) 먼저 합의할 기본 규칙
- 루트 워크스페이스를 기준 소스 오브 트루스로 사용한다.
- 같은 시간에 같은 파일을 둘이 동시에 만지는 것은 원칙적으로 피한다.
- 세션 시작 전 "이번 턴의 소유 범위"를 1줄로 남긴다.
- 세션 종료 전 `무엇을 바꿨는지 / 무엇을 검증했는지 / 다음 사람이 바로 볼 파일`을 남긴다.
- 문서 기준 진입 순서는 `README.md` -> 역할 메모(`CODEX.md`, `CLAUDE.md`) -> 최신 `docs/handover-YYYY-MM-DD.md`다.

## 2) 이 레포에서 충돌이 잘 나는 파일
- `backend/src/index.js`
- `frontend/lesson-booking-app.js`
- `frontend/app.html`
- `frontend/login-flow.js`

위 파일들은 기능이 많이 몰려 있어서 "학생/선생 분업"처럼 도메인을 나눠도 결국 같은 파일에서 충돌할 가능성이 높다.

## 3) 비교적 안전하게 나눌 수 있는 영역
- 학생 전용 화면
  - `frontend/student.html`

  - `frontend/student-bookings.html`

- 선생님 전용 화면
  - `frontend/teacher.html`

  - `frontend/teacher-completed.html`

  - `frontend/power-admin*.html`

- 문서/핸드오버
  - `README.md`
  - `docs/*.md`

## 4) 협업 방식 아이디어

### A. 하루 단위 교대 운영
- 방식: 하루는 Codex, 다음 날은 Claude가 주도권을 잡고 코딩한다.
- 장점:
  - 같은 파일 동시 수정 충돌이 거의 없다.
  - 인수인계 문서 품질이 자연스럽게 올라간다.
  - 한 명이 흐름을 끝까지 잡고 가기 좋다.
- 단점:
  - 급한 기능을 동시에 밀기 어렵다.
  - 문서가 부실하면 다음 날 맥락 복구 비용이 크다.
- 잘 맞는 상황:
  - 대형 리팩토링
  - 백엔드 핵심 로직 수정
  - 상태 전이 정책 정리

### B. 학생/선생 도메인 분리
- 방식: 한 명은 학생 경험, 한 명은 선생님 경험을 맡는다.
- 장점:
  - 화면 시나리오가 비교적 명확하다.
  - 기획 대화도 나누기 쉽다.
- 단점:
  - 실제 구현은 `frontend/lesson-booking-app.js`, `frontend/app.html`, `backend/src/index.js`에 같이 모여 있어 충돌이 쉽게 난다.
- 잘 맞는 상황:
  - 화면 문구/레이아웃/전용 페이지 위주 작업
  - 공통 파일 수정이 적은 작은 개선
- 운영 팁:
  - 공통 파일은 하루에 한 명만 수정한다.
  - 학생/선생 전용 html만 나눠서 만지는 경우에만 추천한다.

### C. 프론트/백엔드 분리
- 방식: 한 명은 프론트, 한 명은 백엔드/API를 맡는다.
- 장점:
  - 가장 전형적이고 이해하기 쉽다.
  - 테스트와 구현을 병렬로 돌리기 좋다.
- 단점:
  - 스펙이 흔들리면 바로 재작업이 생긴다.
  - 백엔드가 `backend/src/index.js` 한 파일이라 백엔드 쪽은 다시 세부 분리가 필요하다.
- 잘 맞는 상황:
  - API 계약이 이미 확정된 기능
  - 관리자 화면처럼 프론트와 백엔드 경계가 분명한 기능

### D. 구현 담당 / 리뷰 담당
- 방식: 한 명이 구현하고, 다른 한 명은 리뷰·리스크 점검·테스트 공백 정리를 맡는다.
- 장점:
  - 같은 파일 충돌이 적다.
  - 회귀 위험이 큰 레포에서 안정적이다.
  - 큰 파일 하나를 빠르게 고칠 때 특히 좋다.
- 단점:
  - 체감 속도는 병렬 구현보다 느릴 수 있다.
- 잘 맞는 상황:
  - `backend/src/index.js`
  - `frontend/lesson-booking-app.js`
  - 인증/권한/시간 계산 로직

### E. 기능 구현 / 테스트·문서 분리
- 방식: 한 명은 기능을 만들고, 다른 한 명은 테스트/문서/핸드오버/가이드 업데이트를 맡는다.
- 장점:
  - 끝날 때 결과물이 깔끔하다.
  - "구현은 됐는데 설명과 검증이 없다"는 문제를 줄인다.
- 단점:
  - 테스트 담당이 기능 의도를 늦게 이해하면 속도가 떨어질 수 있다.
- 잘 맞는 상황:
  - 릴리즈 직전
  - 관리자 기능이나 정책 변경
  - README/가이드/핸드오버 갱신이 중요한 작업

### F. 신규 기능 / 안정화 분리
- 방식: 한 명은 새 기능을 밀고, 다른 한 명은 기존 흐름 버그 수정과 UX 다듬기를 맡는다.
- 장점:
  - 사용자 체감 품질과 기능 개발을 동시에 챙길 수 있다.
  - 충돌 빈도가 비교적 낮다.
- 단점:
  - 새 기능이 공통 파일을 크게 건드리면 다시 얽힐 수 있다.
- 잘 맞는 상황:
  - 새 화면 추가와 기존 로그인/문구/폼 UX 보강이 동시에 필요한 기간

### G. 파워관리자 분리 운영
- 방식: 한 명은 학생/선생 예약 도메인, 다른 한 명은 파워관리자 콘솔만 맡는다.
- 장점:
  - 파일 경계가 가장 명확한 편이다.
  - `frontend/power-admin*.html`, `frontend/power-admin-app.js`, `/api/v1/admin/*`는 따로 묶어서 보기 좋다.
- 단점:
  - 관리자 API도 결국 `backend/src/index.js`에 있으므로 백엔드 충돌은 주의해야 한다.
- 잘 맞는 상황:
  - 운영툴 보강
  - soft delete, 패치노트, 권한정책 같은 작업

### H. 스파이크 / 제품화 분리
- 방식: 한 명은 빠른 프로토타입이나 설계 스파이크, 다른 한 명은 검증된 안을 실제 코드 품질 기준으로 다듬는다.
- 장점:
  - 탐색 속도와 마감 품질을 동시에 가져가기 쉽다.
- 단점:
  - 역할이 흐려지면 중복 구현이 생긴다.
- 잘 맞는 상황:
  - 새로운 예약 정책
  - 소셜 로그인
  - 리팩토링 방향 탐색

## 5) 이 레포에 추천하는 기본 운영 조합

### 추천 1. 평소 기본값
- Codex: 구현 + 마감 + 테스트/리스크 점검
- Claude: 리뷰 + 설계 정리 + 테스트 공백/UX 공백 발견
- 이유:
  - 현재 공통 로직이 큰 파일에 몰려 있어서 둘이 동시에 같은 파일을 크게 고치면 충돌 가능성이 높다.
  - 구현/리뷰 분리가 가장 안전하다.

### 추천 2. 화면 추가가 많은 주간
- Codex: 학생/선생 공통 앱 로직, API 연결
- Claude: 전용 html 페이지, 문구/온보딩/가이드, 파워관리자 화면
- 이유:
  - 공통 JS는 한 명만 잡고, 페이지/문서 쪽은 분리하기 쉽다.

### 추천 3. 큰 기능 주간
- 1일차: Claude가 스파이크/리뷰/리스크 정리
- 2일차: Codex가 구현/정리/검증
- 3일차: Claude가 후속 리뷰 + 테스트 공백 체크
- 이유:
  - 하루 교대와 구현/리뷰 분리를 섞은 방식이 현재 구조에 가장 현실적이다.

## 6) 피하는 게 좋은 방식
- 둘 다 동시에 `backend/src/index.js`를 크게 수정
- 둘 다 동시에 `frontend/lesson-booking-app.js`를 크게 수정
- 학생/선생 분업이라고 해놓고 공통 파일을 둘 다 만지기
- 문서 없이 "내가 조금만 보고 바로 이어서 고칠게" 방식으로 릴레이하기

## 9) 지금 당장 써볼 만한 운영안
1. 평소에는 `구현 담당 / 리뷰 담당`으로 간다.
2. 화면 위주 작업 주간에는 `학생·선생 예약 도메인 / 파워관리자·문서 도메인`으로 나눈다.
3. 큰 기능은 `하루 교대 운영 + 리뷰 문서 첨부` 방식으로 진행한다.
4. 공통 파일은 하루에 한 명만 수정한다.
5. 매 세션 끝에 `docs/handover-YYYY-MM-DD.md` 또는 별도 리뷰 문서에 다음 액션을 남긴다.
